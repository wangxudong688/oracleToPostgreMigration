这本质上是一个 PostgreSQL 的行大小限制问题，和 ora2pg 本身的配置关系不大。下面从原理到解决方案展开。
问题根因
PostgreSQL 默认页大小是 8KB（8192 bytes），减去页头、行头、ItemId 等开销，实际留给一行数据的空间大约在 ~8160 bytes 左右。
一张 900+ 列的表，即便每列数据量不大，累积起来也会超限：
开销项	大小估算
行头（HeapTupleHeader）	~24 bytes
null bitmap（900 列）	ceil(900/8) = 113 bytes
列数据叠加	取决于字段类型
列偏移/长度数组	~1–2 bytes/列

如果每列平均 6–7 bytes（比如一堆 NUMBER、VARCHAR(10)、DATE），900 列就逼近甚至超过 8000 bytes。ora2pg 使用 COPY 方式导入时，PostgreSQL 在构建 tuple 阶段就会报：
row is too large: size ... exceeds maximum ... (8160)
解决方案（按推荐顺序）
1. 垂直拆分（最推荐，架构上正确）
900 列通常意味着表是极度非规范化的设计。按业务语义拆成多个表，用共享主键 1:1 关联：
-- 原表
CREATE TABLE wide_table (
  id NUMBER PRIMARY KEY,
  col1 NUMBER, col2 NUMBER, ... col300 NUMBER,
  col301 VARCHAR2(100), ... col600 VARCHAR2(100),
  ... col900 DATE
);

-- 拆为
CREATE TABLE wide_table_a (
  id NUMBER PRIMARY KEY,
  col1 NUMBER, ... col300 NUMBER
);
CREATE TABLE wide_table_b (
  id NUMBER PRIMARY KEY,
  col301 VARCHAR2(100), ... col600 VARCHAR2(100)
);
CREATE TABLE wide_table_c (
  id NUMBER PRIMARY KEY,
  .......
);
ora2pg 的 FILE_PER_TABLE 和自定义查询（WHERE 配合列白名单）可以分别导出每个子集，再各自导入。
优点： 彻底解决问题，而且 PostgreSQL 查询时跳过无关列，冷数据也可以移到不同的表空间。
2. 降低 toast_tuple_target（无需改表结构，最简单）
PostgreSQL 从 11 开始提供了 toast_tuple_target 存储参数，控制 tuple 达到多大时触发 TOAST 压缩/外存。默认是 2048 bytes。对于宽表，可以设到 128 甚至更低：
ALTER TABLE your_table SET (toast_tuple_target = 128);
如果表还没建，在 ora2pg 的导出 DDL 里加上：
CREATE TABLE your_table (
  ...
) WITH (toast_tuple_target = 128);
原理： 降低阈值后，PostgreSQL 更积极地将列值压缩并移至 TOAST 表中，从而让行在页内变小。但这取决于各个列类型是否能被 TOAST 处理（可 TOAST 的类型：varchar, text, bytea, numeric 等；固定长度类型如 int8, timestamp 不能被 TOAST）。
注意： 如果列大多是 NUMBER / INT / DATE 这类固定长度类型，TOAST 也无法帮上忙。
3. 增大 PostgreSQL 的 block size（需要重编译）
从源码编译 PostgreSQL 时指定 --with-blocksize=32（单位 KB，可选 1, 2, 4, 8, 16, 32），将页大小改为 32KB。这样最大行上限从 ~8KB 升到 ~32KB。
如果不便自行编译，云服务或 Docker 环境可能不支持。且这是全局参数，对更小行的表也会浪费空间和 I/O，不太推荐。
4. 改用 FDW + 查询层透传
不真正导入到 PG，而是通过 oracle_fdw 在 PG 内直接查询 Oracle 的宽表。但只能缓解查询层的问题，数据仍然留在 Oracle。
5. 部分字段用 JSONB/复合类型聚合
把 900 列中语义相近的一组小列聚合成一个 JSONB 或复合类型列：
col_group JSONB  -- {"col1": val1, "col2": val2, ...}
这样在 pg 层面只有少量列，不再触发行大小限制。代价是丢失了列上的索引、约束、列级权限等能力，只适合"内容为主、查询条件多为 id"的场景。
与 ora2pg 相关的配置提示
DATA_LIMIT 控制的是导出文件大小或批次行数，不解决单行超限。
用 TYPE TABLE 导出时，可以在 DDL 里加上 toast_tuple_target。
如果用 COPY 模式导入失败，可以换 INSERT 模式（数据分 INSERT 语句，PG 端行大小判定是一样的，不会绕过限制）。
如果源 Oracle 列非常多，考虑在 EXPORT 的 WHERE 子句里按列分组导出多个文件，分别映射到拆分后的表。
总结
方案	改动量	效果	推荐度
垂直拆分	重构 schema + 适配查询	彻底，数据模型变好	★★★★★
降低 toast_tuple_target	一行 ALTER	立竿见影，但依赖 TOAST 能力	★★★★
增大 block size	重编译 PG	全局生效，部署麻烦	★★★
JSONB 聚合	改 schema 和查询	折衷，损失列级能力	★★
FDW 透查	不改存储	治标不治本	★

如果这些表在下游主要做宽表展示或 OLAP 聚合，垂直拆分是长远方向；如果是生产高频 OLTP，先试 toast_tuple_target = 128 快速绕过，再择机拆分。

方案 1：INSERT 一半列 + UPDATE 另一半
结论：不可行。
为什么不行
PostgreSQL 的 UPDATE 在 MVCC 下不是原地修改，而是写一条全新的 tuple（新行版本）。这条新 tuple 包含所有 900 列的数据。所以：
Step 1: INSERT (col1..col450)  → tuple 1，只有 450 列，可能能放进去
Step 2: UPDATE SET col451..col900 → tuple 2，包含全部 900 列数据
最终落到磁盘上的 tuple 2 和一次性 INSERT 900 列的 tuple 在物理大小+布局上完全一样，同样撞 8160 byte 的上限。
唯一能工作的变体（但已不是同一方案）
如果分两张物理表，各 450 列，共享同一个 PK，然后通过 VIEW + JOIN 对外暴露——这就是垂直拆分，它本身不依赖 INSERT/UPDATE 技巧：
CREATE TABLE t_a (id PK, col1..col450);
CREATE TABLE t_b (id PK, col451..col900);

-- 先插入 t_a
INSERT INTO t_a SELECT id, col1..col450 FROM source;

-- 再插入 t_b（通过 JOIN）
INSERT INTO t_b SELECT id, col451..col900 FROM source;
两张表各自行头减半，各自行宽减半，各自能放下。但这已经不是"同一张表先 INSERT 再 UPDATE"，而是显式垂直分区。
方案 2：PG 也使用 EUC 字符集来减少行宽
结论：技术上可行，但效果有限，且代价较高。
原理验证
格式	CJK 字符	ASCII 字符
UTF-8	3 bytes	1 byte
EUC-CN/EUC-JP/EUC-KR	2 bytes	1 byte

对纯 CJK 文本，EUC 比 UTF-8 省 33%。如果一个 VARCHAR(100) 列存了 100 个中文字：
UTF-8: 300 bytes 数据
EUC: 200 bytes 数据 → 省 100 bytes
但问题在于：900 列的瓶颈不只在文本数据
回顾前面算过的固定开销：
项目	大小
tuple 头	24 bytes
null bitmap (900 列)	113 bytes
变长列偏移数组 (900 列)	~1800 bytes
小计（固定开销）	~1937 bytes
页内可用	8160 - 1937 = 6223 bytes

这 6223 bytes 才是留给实际数据的空间，要分给 900 列。
UTF-8 极限: 6223 / 3 ≈ 2073 个 CJK 字符 / 900 列 ≈ 2.3 字符/列
EUC   极限: 6223 / 2 ≈ 3111 个 CJK 字符 / 900 列 ≈ 3.5 字符/列
EUC 只能多容 ~1 个中文字每列。 如果原始数据里每列本来就只存 1–2 个中文字（如缩写代码、状态标志），那 EUC 的确有可能刚好把行压进 8KB。但如果有任何一列值较长，或者有大量数值/日期类型的列（字符集对它们影响为零），EUC 就杯水车薪。

900 列的情况，TOAST 确实帮不上忙。原因比你想象的要底层。
为什么 TOAST 也没用
TOAST 只对 单列数据超过约 2KB 才触发，把那个列的内容移出去换成 18 字节的指针。但 900 多列的问题出在列元数据本身：
每行 tuple 的固定开销：
  │  HeapTupleHeader (23 字节)
  │  Null bitmap  (900 列 ≈ 113 字节)
  │  变长列偏移数组 (每列 4 字节，如果全是 VARCHAR 那就是 3600 字节)
  │  固定长度列的实际数据（INT / TIMESTAMP / CHAR 等，不能 TOAST）
  │  ...
  ├── 即使每列只占几字节，还没算数据，上半部分已经把 8KB 撑满了
关键点：PostgreSQL tuple 内每列（哪怕值是 NULL）都要在元数据层占位。列数到了 900，即使大多数列都不填数据，Null bitmap + 偏移表 + header 就已经吃掉大量空间，剩下的放不下任何有意义的数据。
可选的解决路径
1. 垂直分表 — 最推荐的做法
这是 PostgreSQL 应对宽表的标准答案。按业务逻辑把列分成几个 100~200 列的组，1:1 关联：
CREATE TABLE t_core (
    id          BIGSERIAL PRIMARY KEY,
    col_001     TEXT,
    col_002     INTEGER,
    ...
    col_200     TIMESTAMP
);

CREATE TABLE t_extended_1 (
    id          BIGINT PRIMARY KEY REFERENCES t_core(id),
    col_201     TEXT,
    col_202     NUMERIC,
    ...
    col_400     VARCHAR(100)
);

CREATE TABLE t_extended_2 (
    id          BIGINT PRIMARY KEY REFERENCES t_core(id),
    col_401     TEXT,
    ...
    col_900     TEXT
);
还原完整视图（只读）：
CREATE VIEW v_full_table AS
SELECT c.*, e1.*, e2.*
FROM t_core c
LEFT JOIN t_extended_1 e1 USING (id)
LEFT JOIN t_extended_2 e2 USING (id);
好处：
每张表都在千列以下，PostgreSQL 的 tuple layout 完全能应付
对应用透明（改视图即可）
后续可以只 JOIN 需要的分表，减少行幅，I/O 也少
代价：
写入需要事务包裹，确保各分表一致
必须通过 JOIN 查询，写操作要在应用层或触发器里做多表 INSERT
2. 把弱关联列合并成 JSONB
如果这 900 列里有大量低频访问或结构相似的列（比如自定义属性、扩展字段），直接合并成一两个 JSONB 列：
CREATE TABLE wide_table (
    id         BIGSERIAL PRIMARY KEY,
    col_001    TEXT,
    col_002    INTEGER,
    ...        -- 核心高频列放出来
    col_100    TIMESTAMP,
    extra      JSONB   -- 剩下的 800 列全塞这里
);
导入时：
INSERT INTO wide_table (id, col_001, ..., col_100, extra)
SELECT
    id, col_001, ..., col_100,
    jsonb_build_object(
        'col_101', col_101,
        'col_102', col_102,
        ...
    ) AS extra
FROM oracle_source;
访问时：
-- 如果是普通查询不要大量解 JSONB，不需要拆
SELECT * FROM wide_table WHERE extra->>'col_500' = 'foo';
JSONB 自身是 TEXT 走 TOAST，单列数据大小不受 8KB 约束。
适合的场景： 列名是 ATTR_001…ATTR_800 这类无明显业务语义、也很少单独建索引的字段。
3. 增大 page size — 最后的物理手段
如果无论如何都不改表结构（比如 Oracle 来的历史系统，应用层 SQL 是 SELECT * + 列序号引用的代码），那就只能：
# 重新编译 PostgreSQL
./configure --with-blocksize=32   # 页面 32KB，单行上限 ~32736 字节

# 然后全库 dump / initdb / reload
与垂直分表的对比：
方案	最大行宽	需改应用	运维代价
垂直分表	不受限（多个 JOIN）	需改写入路径，查询靠视图	低
JSONB 合并	单个 JSONB 不受限	需要改查询语法	低
32KB page	~32736 字节	无需改	极高（全库重建+reload）

16KB 页面下最大行约 16352 字节。32KB 下约 32736 字节。
但 900 列不够——算个账：
即使 32KB page，假设每列平均只占 8 字节（很多类型不止）：
900 列 × 8 字节 = 7200 字节数据
加上 Null bitmap（~113 字节）+ 偏移表 + header = 轻松破万
更别说还有 VARCHAR 长度占位
32KB 放 900 列物理上可能勉强够，但非常不优雅，完全不适合长期维护。

1. 先梳理 900 列的业务归属
