从日志看当前配置的关键问题
对照日志中的失败原因，当前配置中值得关注的点：
PLSQL_PGSQL = 0 — PL/SQL → PL/pgSQL 转换被关闭了，所以所有包/函数/过程的原生语法差异都未被处理，导致 CURSOR ... RETURN、TYPE ... IS TABLE OF、sys_refcursor 等直接报错
PG_SUPPORTS_PARTITION = 0 — 声明式分区被禁用，但本次迁移未走到分区导出步骤所以暂时没触发问题
EXPORT_SCHEMA = 0 + SCHEMA = CHANGE_THIS_SCHEMA_NAME — 这两个值还未改成实际的 schema，说明这份配置是模板尚未根据真实环境调整
如果你后续要修正这些错误，最直接的方式是把 PLSQL_PGSQL 设为 1 让转换引擎工作起来，然后根据生成的 SQL 手工修补转换不完整的部分。

容易混淆的反直觉写法
少数配置项的命名是"否定式"的，需要特别注意反过来理解：
配置项	值	字面理解陷阱	实际含义
NOESCAPE = 0	0	"no-escape = false" → ❌ 好像禁用了转义	反的：启用转义（"no-escape 不为真" = 转义不关闭）
DISABLE_COMMENT = 0	0	"disable-comment = false" → ❌ 好像禁用了注释	反的：不禁止注释 → 实际导出注释
NO_LOB_LOCATOR = 0	0	"no-lob-locator = false" → ❌ 好像没禁用定位器	实际上值 0 表示不使用 LOB 定位器（全量加载）

你日志里看到的核心问题
PLSQL_PGSQL = 0 是日志中包(PACKAGE)、函数(FUNCTION)、过程(PROCEDURE)转换失败的根本原因——它把 PL/SQL → PL/pgSQL 转换引擎完全关闭了，生成的 package.sql、function.sql、procedure.sql 保留的是几乎原样的 Oracle 语法，psql 执行时自然报错。
如果把这些全改成 1，转换引擎会工作，但转换质量取决于 ora2pg 版本和代码复杂度，仍需要手工验证。
