这是一个很实际的迁移验证问题。整体思路是：把两边元数据分别提取成可比较的文本或结构化数据，然后做结构化对比。下面说几种方法，以及完整的验证过程。
方法选型
比较好的方案是 元数据导出 + 结构化比对。具体有两种路线：
方法	适用场景	精度
SQL 脚本交叉查询（dblink / 外部表）	两边可直连，且 Oracle 端允许只读访问	精确到每列每约束
元数据导出 + diff（pg_dump --schema-only vs Oracle DBMS_METADATA）	不能直连，或要留审计记录	依赖导出格式，需后处理

如果不能直连两边，或要留下可审计的比对报告，推荐第二种。下面以 导出 + diff 路线为主展开。
完整比对过程
第一阶段：导出基准清单（从 Oracle）
在 Oracle 端运行一组查询，输出所有对象的规范清单。把下面查询导出为 CSV：
-- 1. 表及列信息
SELECT
    owner,
    table_name,
    column_name,
    data_type || '(' || data_length || ')' AS data_type,
    nullable,
    data_default
FROM all_tab_columns
WHERE owner = 'YOUR_SCHEMA'
ORDER BY table_name, column_id;

-- 2. 主键与约束
SELECT
    owner, table_name, constraint_name, constraint_type,
    column_name, position
FROM all_cons_columns
JOIN all_constraints USING (owner, constraint_name, table_name)
WHERE owner = 'YOUR_SCHEMA'
ORDER BY table_name, constraint_name, position;

-- 3. 索引
SELECT
    owner, table_name, index_name, uniqueness,
    column_name, column_position
FROM all_ind_columns
JOIN all_indexes USING (owner, index_name)
WHERE owner = 'YOUR_SCHEMA'
ORDER BY table_name, index_name, column_position;

-- 4. 序列
SELECT sequence_owner, sequence_name,
       min_value, max_value, increment_by,
       last_number
FROM all_sequences
WHERE sequence_owner = 'YOUR_SCHEMA';

-- 5. 视图（只比对定义哈希，不比全文，否则噪音太大）
SELECT owner, view_name, DBMS_OBFUSCATION_TOOLKIT.md5(text) AS md5
FROM all_views
WHERE owner = 'YOUR_SCHEMA';
每条 CSV 输出成一个独立文件：ora_tables.csv, ora_constraints.csv, ora_indexes.csv, ora_sequences.csv, ora_views.csv。
第二阶段：导出基准清单（从 PostgreSQL）
在 PostgreSQL 端用 information_schema + pg_catalog 提取等价的元数据，保持列顺序一致以便 diff。

    column_default

-- 2. 主键与约束

    tc.constraint_name, tc.constraint_type,
    kcu.column_name, kcu.ordinal_position
FROM information_schema.table_constraints tc
JOIN information_schema.key_column_usage kcu
  ON tc.constraint_name = kcu.constraint_name
 AND tc.table_schema   = kcu.table_schema

-- 3. 索引

    schemaname, tablename, indexname, indexdef

WHERE schemaname = 'public'

-- 4. 序列
SELECT sequence_schema, sequence_name,
       start_value, minimum_value, maximum_value, increment
FROM information_schema.sequences
WHERE sequence_schema = 'public';

-- 5. 视图定义哈希
SELECT
    table_schema, table_name,
    MD5(view_definition) AS md5
FROM information_schema.views
WHERE table_schema = 'public';
同样输出 CSV：pg_tables.csv, pg_constraints.csv, pg_indexes.csv, pg_sequences.csv, pg_views.csv。
第三阶段：结构化 diff
# 对每个维度做 diff

diff <(sort ora_constraints.csv) <(sort pg_constraints.csv) > diff_constraints.txt
diff <(sort ora_indexes.csv) <(sort pg_indexes.csv) > diff_indexes.txt
diff <(sort ora_sequences.csv) <(sort pg_sequences.csv) > diff_sequences.txt
diff <(sort ora_views.csv) <(sort pg_views.csv) > diff_views.txt
注意：Oracle 和 PostgreSQL 的数据类型名称不同（VARCHAR2 → character varying, NUMBER(10,0) → integer 等）。建议在导出时做一层映射，或者比对的容差规则里允许已知的等价类型对。一个实用做法：在 Oracle 导出脚本里用 CASE 或者用 Python 脚本对两边做 normalize 后再 diff。

工具	定位	输出

总结建议

需要补充验证的对象
1. 函数 / 存储过程 / 包
Oracle 的包（Package）会被 Ora2Pg 或手动拆成 PostgreSQL 的 schema + 独立函数，迁移过程中最容易出问题。
-- Oracle 端
SELECT owner, object_name, object_type,
       DBMS_OBFUSCATION_TOOLKIT.md5(DBMS_METADATA.GET_DDL(object_type, object_name)) AS md5
FROM all_objects
WHERE owner = 'YOUR_SCHEMA'
  AND object_type IN ('FUNCTION', 'PROCEDURE', 'PACKAGE', 'PACKAGE BODY');

-- PostgreSQL 端
SELECT n.nspname AS schema_name,
       p.proname AS function_name,
       pg_get_functiondef(p.oid) AS definition
FROM pg_proc p
JOIN pg_namespace n ON p.pronamespace = n.oid
WHERE n.nspname = 'public';
注意：Oracle 包没有直接对应物，所以不能简单比 MD5。建议比三样：
函数名 + 参数签名 是否完整迁移
包内公有函数是否全部拆出
包体私有函数是否处理好（一般转为 schema 内私有函数或标记 internal）

2. 触发器
-- Oracle
SELECT owner, trigger_name, table_name,
       triggering_event, status, trigger_type
FROM all_triggers
WHERE owner = 'YOUR_SCHEMA';

-- PostgreSQL
SELECT event_object_schema, event_object_table,
       trigger_name, event_manipulation,
       action_timing, action_orientation
FROM information_schema.triggers
WHERE trigger_schema = 'public';
重点检查：
触发器是否全部创建成功
触发时机（BEFORE/AFTER/INSTEAD OF）是否对应
行级触发器（FOR EACH ROW）是否一致
被禁用的触发器状态是否保留
3. 物化视图
-- Oracle
SELECT owner, mview_name, refresh_mode, refresh_method, last_refresh_type
FROM all_mviews
WHERE owner = 'YOUR_SCHEMA';

-- PostgreSQL
SELECT schemaname, matviewname,
       is_populated,
       pg_size_pretty(pg_total_relation_size(schemaname||'.'||matviewname)) AS size
FROM pg_matviews
WHERE schemaname = 'public';
关键点：刷新模式（COMPLETE / FAST / FORCE）在 PostgreSQL 中只有 REFRESH MATERIALIZED VIEW CONCURRENTLY，Ora2Pg 迁移时可能降级为全量刷新。
4. 自定义类型
Oracle 的 OBJECT TYPE / VARRAY / NESTED TABLE → PostgreSQL 的 composite type / 数组 / 自定义类型。
-- Oracle
SELECT owner, type_name, typecode
FROM all_types
WHERE owner = 'YOUR_SCHEMA';

-- PostgreSQL
SELECT n.nspname, t.typname, t.typtype
FROM pg_type t
JOIN pg_namespace n ON t.typnamespace = n.oid
WHERE n.nspname = 'public'
  AND t.typtype IN ('c', 'd', 'e');  -- composite, domain, enum
5. 同义词
Oracle 的 SYNONYM 在 PostgreSQL 里没有直接等价物。迁移时一般用视图或 search_path 模拟。需要逐个确认每个 synonym 在 PostgreSQL 端有对应的访问路径。
-- Oracle
SELECT owner, synonym_name, table_owner, table_name, db_link
FROM all_synonyms
WHERE owner = 'YOUR_SCHEMA';
对比项：所有 PUBLIC 和 PRIVATE 同义词是否都有对应的处理方案。
6. 表分区
Oracle 的分区语法（RANGE / LIST / HASH / 复合分区）与 PostgreSQL 的分区语法不一一对应。
-- Oracle
SELECT owner, table_name, partitioning_type, subpartitioning_type,
       partition_count, status
FROM all_part_tables
WHERE owner = 'YOUR_SCHEMA';

-- PostgreSQL (pg12+)
SELECT
    p.relname AS table_name,
    pg_get_expr(c.relpartbound, c.oid) AS partition_bound
FROM pg_class p
JOIN pg_inherits i ON p.oid = i.inhrelid
JOIN pg_class c ON i.inhparent = c.oid
JOIN pg_namespace n ON c.relnamespace = n.oid
WHERE n.nspname = 'public'
  AND c.relispartition = false;
常见问题：Oracle 的 INTERVAL 分区在 PostgreSQL 里没有直接等价物，需转为 range 分区 + 自动创建分区的函数/触发器模拟。
7. 注释（COMMENT）
很多人忽略这个。
-- Oracle

FROM all_col_comments
WHERE owner = 'YOUR_SCHEMA';

-- PostgreSQL

FROM pg_description pgd

  ON pgd.objoid = c.table_name::regclass

8. 权限 / 角色 / 授权
Oracle 的 GRANT 模型和 PostgreSQL 差异很大（Oracle 有系统权限/对象权限/角色层级，PostgreSQL 用 default privileges + role inheritance）。
这个维度人工核对为主，建议对照清单逐项确认：
检查项	Oracle 端	PostgreSQL 端
用户/角色是否存在	dba_users	\du / pg_roles
表级别 SELECT/INSERT/UPDATE/DELETE 权限	dba_tab_privs	\dp / information_schema.table_privileges
默认权限（新对象自动继承）	无直接等价物	ALTER DEFAULT PRIVILEGES
执行权限（函数/过程）	dba_tab_privs WHERE TYPE='EXECUTE'	\df+ / information_schema.routine_privileges

JOIN all_indexes idx

 AND ic.index_name   = idx.index_name
