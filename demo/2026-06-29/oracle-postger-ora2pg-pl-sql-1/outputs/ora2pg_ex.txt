这个问题在 `ora2pg` 迁移 Oracle Package（包）时非常常见。**这是 ora2pg 工具的一个已知局限**：它试图将 Oracle 的 **Package Spec（规范）**直接转换为 DDL，但 PostgreSQL 根本不支持 "全局公共游
标" (`GLOBAL CURSOR`) 或 `%ROWTYPE`。

在 PG 中，所有逻辑必须封装在 `FUNCTION`, `PROCEDURE` 或者匿名块内部，不能像 Oracle 那样单独定义一个 `CURSOR ... RETURN TYPE;`。

针对这个错误，既然您的目标是尽快跑通数据导入（Data First），我建议采用以下 **"两步走"** 策略来解决：

### 🚀 方案一：跳过包逻辑优先导入表结构（最快解决）
如果您现在的重点是 **把表和数据结构导进去**，Oracle Package 中的游标声明和 `%ROWTYPE` 通常只是业务接口定义，不直接影响数据存储。

#### ✅ 操作步骤：
1.  **临时重命名该文件**：避免 `import_all.sh` 再次执行报错阻断流程。
    ```bash
    mv ./schema/packages/package.sql ./schema/packages/package.sql.bak
    # 或者在 import_all.sh 中注释掉该行调用
    # psql -f ./schema/packages/package.sql -> # psql ... (注释)
    ```

2.  **继续运行导入脚本**：确保 `import_all.sh` 只执行表结构（Tables）和数据。

3.  **后续处理逻辑**：等所有数据都进 Postgre 后，再针对业务代码去重写存储过程/函数。此时您可以手动将 `%ROWTYPE` 替换为具体的列名或自定义类型。

---

### 🛠️ 方案二：修复 `package.sql` (如果您必须保留它)
如果您必须要在第一步包含这个文件（例如脚本逻辑强制依赖），您需要使用正则表达式清理这段非法 SQL，或者配置 ora2pg 的生成规则。

#### 🔧 方法 A: 直接注释掉游标声明行
既然报错在 `line 15` 附近且是 `%ROWTYPE` 这种 Oracle 特有语法，最快的方法是把这一整块包定义都视为“业务逻辑”暂时忽略，只保留必要的权限或表结构。

使用 `sed` 快速处理该文件：
```bash
# 备份并替换掉非法的 ORACLE CURSOR %ROWTYPE 行（仅作为临时修复）
cd ./schema/packages/
cp package.sql package.sql.backup
# 删除包含 "%ROWTYPE" 的行，或者将 "CREATE PACKAGE ..." 相关的声明注释掉
sed -i 's/%CURSOR.*/-- &/' package.sql   # 尝试匹配 CURSOR
# 更稳妥的做法：直接将整个 Package Spec (如果是纯 spec) 的 DDL 全部注释或移除
```

**最推荐的直接修复命令（针对报错行）：**
由于 `ora2pg` 生成的 PG SQL 有时不会自动把 Oracle Cursor 转成 Function，建议直接用以下脚本删除这一条非法语句：
```bash
# 进入包目录
cd ./schema/packages/

# 创建一个新文件处理过的版本
awk '/^CURSOR.*RETURN.*/{next} {print}' package.sql > package_fixed.sql && mv package_fixed.sql package.sql

echo "已移除 CURSOR ... RETURN %ROWTYPE; 行"
```

### ⚙️ 方案三：正确配置 `ora2pg` (一劳永逸)
如果您不想每次都手动改文件，说明 ora2pg 的配置或参数可能没有正确处理 Package。PostgreSQL **不支持 Oracle 的 Package Spec**（只有 Function/Procedure）。

1.  **检查生成模式**：
    在运行 `ora2pg` 时，确保明确告诉它把包转成函数 (`-t function`) 而不是保留为 `-c package` (这通常只用于结构导出)。
    
    *错误示例*:
    ```bash
    # 如果直接导出了 create object type package ...
    ora2pg -D pgdb.conf -o schema.sql 
    ```

    *推荐做法*：让 `ora2pg` 把包逻辑转成 PG **函数**。
    ```bash
    # 生成时指定将 Package Body 转为 Function/Procedure
    ora2pg -c ORACLE_CONFIG.txt \
           -t package \
           --force-package-to-function \ 
           > ./schema/packages/package.sql
    
    # ⚠️ 注意：ora2pg 默认参数可能因版本不同而异。检查您的 config file (orapg.conf)
    ```

2.  **修改 `config` 文件** (`ora2pg.config`)：
    找到或添加这一行，强制转换逻辑（如果您的 ora2pg 支持该选项）：
    *   **DB_PACKAGE_TO_FUNCTION**: `yes` (默认通常是 yes，但可能受限于代码结构)。

3.  **解决 `%ROWTYPE`**：
    PG 不支持 `%ROWTYPE`。如果必须保留这个返回类型定义，ora2pg 应该自动将其扩展为具体的列名列表（如 `(id INT, name TEXT...)`）。如果它没转过来，说明 ora2pg 未能关联到 `emp` 表的最新结构解
析信息。
    
    **手动修复 `%ROWTYPE`**:
    ```sql
    -- ❌ Oracle / Bad PG: CURSOR ... RETURN emp%ROWTYPE;
    -- ✅ PostgreSQL (必须在函数内):
    CREATE OR REPLACE FUNCTION get_high_earners() 
    RETURNS SETOF RECORD AS $$
       BEGIN
          RETURN QUERY SELECT * FROM public.emp;
       END;
    $$ LANGUAGE plpgsql;
    ```

2.  **后期处理**：等 Postgre SQL 建好表和导入完数据后，再单独检查那个包里的逻辑（业务代码），手动重写为 PG Function。

#!/bin/bash

target_dir="${1:-.}"

for item in "$target_dir"/*; do
    if [ -d "$item" ]; then
        folder_name="$(basename "$item")"
        file_count=$(ls -1 "$item" 2>/dev/null | wc -l)

        if [ "$file_count" -eq 0 ]; then
                echo "$folder_name nothing"
                continue
        fi
        for file in "$item"/*; do
            if [ -f "$file" ]; then
                echo "$folder_name $(basename "$file")"
            fi
        done
    fi
done



以下是对每一类未处理或出错的类型，从 Oracle 迁移到 PostgreSQL 时需要关注的核心语法差异总结。
一、已处理但失败的 5 种类型
1. PACKAGE / PACKAGE BODY
Oracle	PostgreSQL 注意事项
CREATE PACKAGE pkg AS ... END;	PostgreSQL 没有包(package)概念，需将包拆分为独立的函数/过程，或放在同一 schema 下用命名前缀模拟
包内游标 CURSOR c RETURN table%ROWTYPE	PostgreSQL 不支持 CURSOR ... RETURN，需改为 CURSOR FOR SELECT * FROM table 或用 REFCURSOR 返回
包全局变量 / 常量	需改为 GUC 自定义参数、临时表或函数内的默认参数
包初始化块	无等价物，需显式调用初始化函数

2. VIEW（涉及 Oracle 系统表/同义词的视图）
Oracle	PostgreSQL 注意事项
引用 sys.all_*, dba_*, user_* 内置视图	PostgreSQL 有 pg_catalog / information_schema 体系，没有直接对应，需 改写查询来源
SELECT ... FROM sys.all_scheduler_program_args	无对应系统表，需手动在 PostgreSQL 中创建等效视图或彻底移除该引用
视图中使用 Oracle 特有函数（DECODE, NVL, SYSDATE, CONNECT BY 等）	需替换为 PostgreSQL 实现（CASE, COALESCE, NOW(), WITH RECURSIVE）

3. FUNCTION（PL/SQL 语法差异）
Oracle	PostgreSQL 注意事项
sys_refcursor 返回类型	改为 refcursor（PostgreSQL 的游标类型）
RETURN 子句中声明游标结构	PostgreSQL 函数返回 refcursor 时需要 $body$ 内 OPEN cursor FOR SELECT ... 并返回该游标
%TYPE 和 %ROWTYPE 属性	仅有限支持，%TYPE 可用但需注意; %ROWTYPE 在 PL/pgSQL 中支持，但位置有限制
IS 替代 AS 后跟函数体	PostgreSQL 需使用 LANGUAGE plpgsql 且函数体用 $$ ... $$ 或 $body$ 包裹
内置函数差异：SYSDATE→NOW(), NVL→COALESCE, DECODE→CASE...	大量内建函数需要逐行替换

4. PROCEDURE
Oracle	PostgreSQL 注意事项
TYPE t_name IS TABLE OF table%TYPE	PostgreSQL 不支持在过程内部定义嵌套表类型，需改为 CREATE TYPE 在 schema 级别创建，或用 ARRAY + 临时表模拟
TYPE t_name IS RECORD (...)	同上，需在 schema 级别 CREATE TYPE。若只在该过程内使用，也可以用 RECORD 类型 + 匿名块
IS / AS 声明后直接写声明节	PostgreSQL 用 DECLARE 开始声明节
FORALL / BULK COLLECT INTO	PostgreSQL 不直接支持，需改为 FOREACH ... ARRAY 或 UNNEST + 批量 INSERT/UPDATE
EXCEPTION WHEN OTHERS THEN ...	PostgreSQL 异常处理语法类似但细节不同（WHEN OTHERS THEN 支持，但不能 SQLCODE 等方式使用）
COMMIT / ROLLBACK 在过程中	PostgreSQL 不允许存储过程内部随意提交/回滚事务（除非在事务块顶层）
Oracle OUT 参数多种绑定	基本兼容但要注意模式: OUT / IN OUT 支持，默认参数只在末位使用

5. INDEX（ora2pg 转换 B-tree / 函数索引的问题）
Oracle	PostgreSQL 注意事项
CREATE INDEX idx ON table (expression DESC)	注意 DESC 不需要也不应该用双引号包裹；正确的写法是 (sal DESC)
函数索引 (UPPER(name))	兼容，但注意 PostgreSQL 函数索引需要在创建时就确定 IMMUTABLE
位图索引 BITMAP	PostgreSQL 不支持位图索引，需改为 B-tree 或自行评估更优策略
反向键索引 REVERSE	不支持，需考虑其他等价方案
基于分区的本地/全局索引	PostgreSQL 分区索引语法不同（CREATE INDEX ON ONLY parent 等）

二、未处理的 7 种类型——迁移注意事项
6. TYPE（自定义类型）
Oracle	PostgreSQL 注意事项
CREATE OR REPLACE TYPE t_name AS OBJECT (...)	PostgreSQL 用 CREATE TYPE t_name AS (...)，但**无方法(member function)**支持（需拆为函数+类型）
CREATE TYPE t_name AS TABLE OF ...	无直接等价，改为 CREATE TYPE ... AS (...) + ARRAY，或用临时表
CREATE TYPE t_name AS VARRAY(...)	PostgreSQL 可用 CREATE TYPE + ARRAY[n]，但不严格限制长度
类型继承 / 超类型	PostgreSQL CREATE TYPE 不支持对象继承

7. GRANT（权限）
Oracle	PostgreSQL 注意事项
GRANT EXECUTE ON pkg TO user	允许（针对函数/过程），但需逐对象授权，无法像包一样整组授予
GRANT CONNECT, RESOURCE, DBA TO user	PostgreSQL 使用 Role + INHERIT / GRANT 权限，无等价角色层级
GRANT SELECT ON schema.table TO role	基本兼容，但 PostgreSQL 默认未授予 PUBLIC 对 schema 的 USAGE 权限
同义词对象、目录的权限	PostgreSQL 不支持 DIRECTORY 和 SYNONYM，对应权限概念也不同

8. TABLESPACE
Oracle	PostgreSQL 注意事项
CREATE TABLESPACE ts DATAFILE '...' SIZE 100M	语法完全不同：CREATE TABLESPACE ts LOCATION '/path'（使用文件系统目录而非数据文件）
自动扩展 AUTOEXTEND ON	不支持，需操作系统层管理空间
BIGFILE / SMALLFILE	不支持，PostgreSQL 表空间只是目录
CREATE TABLE t TABLESPACE ts	语法兼容，但 PostgreSQL 表空间主要用于 I/O 隔离而非存储管理

9. PARTITION
Oracle	PostgreSQL 注意事项
声明式分区语法不同：Oracle PARTITION BY RANGE (col) (PARTITION p1 VALUES LESS THAN (100))	PostgreSQL 语法类似但略有差异，需要 PARTITION BY RANGE (col) + CREATE TABLE p1 PARTITION OF t FOR VALUES FROM (...) TO (...)
列表分区 VALUES (...)	兼容，但写法定型略有不同
哈希分区 PARTITION BY HASH	PostgreSQL 11+ 支持，语法不同
间隔分区 PARTITION BY RANGE ... INTERVAL	PostgreSQL 不支持间隔分区，需使用 pg_partman 扩展
子分区	不支持自动子分区，需手动分层创建
分区修剪与全局索引	行为差异较大，需仔细测试

10. DBLINK
Oracle	PostgreSQL 注意事项
CREATE DATABASE LINK dblink CONNECT TO ...	PostgreSQL 使用 postgres_fdw 或 dblink 扩展实现。语法完全不同：先 CREATE SERVER → CREATE USER MAPPING → CREATE FOREIGN TABLE / IMPORT FOREIGN SCHEMA
SELECT * FROM table@dblink	需改为 SELECT * FROM foreign_table（FDW 方式）或 SELECT * FROM dblink('...') AS (...)
远程 DML INSERT INTO table@dblink	PostgreSQL 需要 FDW 支持写操作（postgres_fdw 默认支持）
同义词引用远程对象	需改造为 FDW 的 foreign table

11. SYNONYM
Oracle	PostgreSQL 注意事项
CREATE SYNONYM s FOR schema.table	PostgreSQL 没有同义词，替代方案：使用 VIEW 模拟（只读场景），或设置 search_path 让名称解析按顺序查找 schema
公共同义词 PUBLIC SYNONYM	无等价概念，可将对象放在 public schema 中，或使用 search_path
同义词指向远程对象	需改为 FDW 的 foreign table

12. DIRECTORY
Oracle	PostgreSQL 注意事项
CREATE DIRECTORY dir AS '/path'	PostgreSQL 没有 DIRECTORY 对象，函数如 COPY, pg_read_file() 直接使用操作系统路径，权限由 pg_read_server_files 等角色属性控制
在 PL/SQL 中用 BFILENAME('DIR', 'file')	无等价，需改为 pg_read_binary_file() + 应用层处理
外部表 ORACLE_LOADER / ORACLE_DATAPUMP	改为 file_fdw 或 COPY 命令
